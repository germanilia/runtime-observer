Answer: are the DB queries duplicates?¶

They are actual backend DB calls captured by instrumentation, not a UI duplication bug. They look similar because SQLAlchemy is loading many relationships for the same agent id. The UI should group them as relationship-loader fan-out and still allow raw-event inspection.

⇋Tradeoffs¶

Store retention pins for relevant logs/traces
Pros
  • Prevents losing context around errors
  • Makes cleanup deterministic
Cons
  • Adds schema and cleanup complexity
  • Needs caps to prevent storage growth
Client-side sortable tables first
Pros
  • Fast to implement
  • No extra server round trips for small bounded lists
Cons
  • Server-side sorting may be needed later for very large lists

◇Key decisions¶

◇ Keep raw events available while adding grouped explanations to avoid hiding details.
◇ Use bounded retention defaults to preserve debugging context without unbounded storage growth.
◇ Enhance embedded dashboard incrementally instead of replacing frontend architecture.

⚠Risks¶

⚠ Cleanup deletes useful investigation data — mitigation: Build protected sets and test exact/nearby error logs.
⚠ SQLite/Postgres migration mismatch — mitigation: Keep schema additive and test both paths where possible.
⚠ Dashboard becomes too complex as a single HTML string — mitigation: Introduce small JS helpers for sortable tables, drawer sizing, and cards.

✕Out of scope¶

✕ Fixing the source app duplicate slug race in this repository.
✕ Replacing Runtime Observer with OpenTelemetry or an external tracing backend.
✕ Building a new React/Vite dashboard app in this phase.
